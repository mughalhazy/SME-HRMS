# API Standards

This document defines the canonical Service Contract Standard (SCS) for SME-HRMS services and the shared API gateway.

## 1) Base path and routing

### Route namespace
- All public routes are versioned under `/api/v1`.
- Service-specific route groups align to the API gateway registry:
  - `/api/v1/employees`, `/api/v1/departments`, `/api/v1/roles`
  - `/api/v1/performance/*`
  - `/api/v1/attendance`
  - `/api/v1/leave`
  - `/api/v1/payroll`
  - `/api/v1/hiring`
  - `/api/v1/auth`
  - `/api/v1/notifications`

### Resource naming
- Use nouns, not verbs, for resource paths.
- Use lowercase kebab-case for path segments.
- Use snake_case for query-parameter names to match the current canonical docs.
- Keep nesting shallow; favor resource IDs over deeply nested paths.

## 2) Request envelope

All service entry points must accept a request envelope with correlation metadata.

```json
{
  "request_id": "uuid",
  "tenant_id": "string",
  "actor": {
    "id": "string",
    "type": "user | service"
  },
  "timestamp": "ISO8601"
}
```

### Request rules
- `request_id` is mandatory and must be propagated end-to-end.
- `tenant_id` is mandatory for multi-tenant workflows.
- `actor.id` and `actor.type` are required for auditability and authorization decisions.
- `timestamp` must be ISO8601 in UTC when generated by the platform.

## 3) Response envelope

All responses must use the same top-level structure.

```json
{
  "status": "success | error",
  "data": {},
  "meta": {
    "request_id": "",
    "timestamp": "",
    "pagination": {}
  },
  "error": null
}
```

### Success rules
- `status` must be `success`.
- `data` contains the resource payload or list payload.
- `meta.request_id` is required on every response.
- `meta.timestamp` is required on every response.
- `meta.pagination` must always be present, using `{}` for non-list responses.

### Error rules
- `status` must be `error`.
- `data` must be an empty object.
- `error` must use the canonical error shape below.
- `meta.request_id`, `meta.timestamp`, and `meta.pagination` remain mandatory on errors.

```json
{
  "status": "error",
  "data": {},
  "meta": {
    "request_id": "",
    "timestamp": "",
    "pagination": {}
  },
  "error": {
    "code": "STRING_CODE",
    "message": "Readable message",
    "details": {}
  }
}
```

## 4) Status codes

### Success
- `200 OK`: successful read/update/action with body.
- `201 Created`: successful resource creation.
- `202 Accepted`: accepted for asynchronous processing.
- `204 No Content`: successful request with no body.

### Client errors
- `400 Bad Request`: malformed request shape.
- `401 Unauthorized`: missing or invalid authentication.
- `403 Forbidden`: authenticated but not authorized.
- `404 Not Found`: resource does not exist.
- `409 Conflict`: unique or state conflict.
- `422 Unprocessable Entity`: validation or business rule failure.
- `429 Too Many Requests`: rate limit exceeded.

### Server and dependency errors
- `500 Internal Server Error`: unexpected service failure.
- `502 Bad Gateway`: upstream dependency returned an invalid response.
- `503 Service Unavailable`: temporary outage or circuit open.
- `504 Gateway Timeout`: upstream dependency timeout.

## 5) Pagination, filtering, and sorting

### Pagination
- Cursor pagination is the default for list endpoints.
- `meta.pagination` is the only allowed location for pagination metadata.
- Supported parameters:
  - `limit` default `25`, max `100`
  - `cursor` opaque continuation token
  - `next_cursor` opaque continuation token for the next page
  - `count` result count for the current page

### Filtering
- Use exact, documented query parameters such as `status`, `department_id`, `employee_id`, `period_start`, `period_end`.
- Use `_from` and `_to` suffixes for range boundaries.

### Sorting
- Use `sort` with comma-separated field names.
- Prefix descending fields with `-`.
- Sort order must be deterministic and documented per endpoint.

## 6) Validation, QC, and auto-fix

### Validation
- Validate enums and state transitions against canonical domain rules.
- Return `422` for business-rule violations such as invalid workflow transitions.
- Return `409` for duplicates or optimistic-concurrency conflicts.

### QC engine
- `all_endpoints_follow_response_format`
- `no_raw_json_responses`
- `error_format_consistent`
- `pagination_standard_applied`
- `meta_fields_present`

### Auto-fix and re-QC
- Services should centralize response wrapping in middleware/helpers rather than formatting envelopes inline everywhere.
- Any contract drift should trigger auto-fix, then re-QC before release.

## 7) Authentication and authorization
- Use bearer tokens in `Authorization: Bearer <token>`.
- All endpoints are authenticated by default unless explicitly public.
- Enforce both capability and scope checks.

## 8) Observability and correlation
- Accept and propagate `X-Trace-Id` or `X-Request-Id` when provided.
- Map the propagated correlation identifier to `meta.request_id` in responses.
- Emit audit records for privileged writes, approvals, payroll actions, access changes, and notification preference changes.

## 9) Versioning and compatibility
- Use major versions in the URL: `/api/v{major}`.
- Breaking changes to the envelope require a new major version or an explicitly coordinated migration window.

## 10) Rejection rule
- Reject builds when any endpoint is not compliant.
- Reject builds when response formatting is inconsistent across services.
